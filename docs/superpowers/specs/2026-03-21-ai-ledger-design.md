# AI 记账 App（SmartLedger）— 产品设计文档

## 1. 产品定位

离线优先的对话式 AI 记账 App。以聊天窗口为统一入口，支持语音、拍照、文字三种记账方式。面向国内安卓市场（小米/华为等应用商店）。

### 核心原则

1. **离线是首要要求** — 手动录入和文字记账在断网时必须完全正常工作，不是降级体验
2. **本地规则引擎优先** — 覆盖 80% 日常记账场景，GLM 只处理规则搞不定的复杂情况
3. **对话式统一入口** — 一个聊天窗口 = 所有记账入口，类似豆包/千问的交互模式

## 2. 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 前端框架 | React Native + Expo (SDK 55) | 跨平台，学习中 |
| 本地数据库 | expo-sqlite | 离线优先，关系型，SQL 熟悉 |
| 本地规则引擎 | 纯 TypeScript（正则 + 关键词匹配） | 零依赖，离线可用，毫秒级响应 |

| OCR（票据识别） | 腾讯云票据识别 | 小票/发票专用 OCR |
| LLM | GLM-4.7-Flash（智谱 AI） | 免费 API，30B MoE 模型 |
| BaaS（云同步） | 腾讯云开发 CloudBase（MySQL） | 统一腾讯生态，国内合规 |
| 图表 | react-native-chart-kit 或 Victory Native | 饼图/折线图 |
| 币种 | CNY / USD | 双币种支持 |

## 4. 核心数据模型

### Account（账户）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT (UUID) | 主键 |
| name | TEXT | 账户名称 |
| type | TEXT | 枚举：wechat / alipay / cash / bank_card |
| currency | TEXT | 枚举：CNY / USD |
| initial_balance | REAL | 初始余额（创建时设定，默认 0） |
| icon | TEXT | 图标标识 |
| sort_order | INTEGER | 排序 |
| deleted_at | TEXT (ISO 8601) | 软删除时间（NULL 表示未删除） |
| created_at | TEXT (ISO 8601) | 创建时间 |
| updated_at | TEXT (ISO 8601) | 更新时间 |

> 当前余额 = initial_balance + SUM(income) - SUM(expense)，通过查询计算得出，不存储。

### Category（分类，预设固定，暂不支持自定义）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT (UUID) | 主键 |
| name | TEXT | 分类名称 |
| icon | TEXT | 图标标识 |
| type | TEXT | 枚举：income / expense |
| is_system | INTEGER | 1 = 系统预设，0 = 用户自定义（预留） |
| sort_order | INTEGER | 排序 |

预设分类：

- **支出**：餐饮、交通、购物、住房、娱乐、医疗、教育、通讯、日用、其他
- **收入**：工资、兼职、投资、红包、其他

> 当前阶段所有分类 is_system = 1，用户自定义分类作为未来扩展预留。

### Transaction（账单）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT (UUID) | 主键 |
| amount | REAL | 金额（正数） |
| currency | TEXT | 枚举：CNY / USD |
| type | TEXT | 枚举：income / expense |
| category_id | TEXT | 外键 → Category |
| account_id | TEXT | 外键 → Account |
| note | TEXT | 备注（可为空） |
| date | TEXT (YYYY-MM-DD) | 账单日期（用户指定，默认当天） |
| source | TEXT | 来源：manual / text / voice / ocr |
| sync_status | TEXT | 同步状态：pending / synced / failed |
| device_id | TEXT | 设备标识（用于多设备同步冲突检测） |
| deleted_at | TEXT (ISO 8601) | 软删除时间（NULL 表示未删除） |
| created_at | TEXT (ISO 8601) | 创建时间 |
| updated_at | TEXT (ISO 8601) | 更新时间 |

> 币种约束：Transaction.currency 必须与关联 Account.currency 一致，在 DAO 层校验。

### ChatMessage（聊天记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT (UUID) | 主键 |
| role | TEXT | 枚举：user / assistant |
| content | TEXT | 消息内容 |
| content_type | TEXT | 枚举：text / voice / image / card |
| parse_status | TEXT | 解析状态：success / fallback / pending / error（仅 assistant 消息） |
| transaction_id | TEXT | 关联账单（可为空，card 类型时关联） |
| created_at | TEXT (ISO 8601) | 创建时间 |

### ParsedTransaction（规则引擎/AI 的统一输出格式）

> 非数据库表，是 TypeScript 接口，作为规则引擎和 AI 解析的统一输出契约。

```typescript
interface ParsedTransaction {
  amount: number;
  currency: 'CNY' | 'USD';
  type: 'income' | 'expense';
  category_id: string | null;    // null 时需用户手动选择
  note: string | null;
  confidence: 'high' | 'low';   // high = 金额+分类都匹配，low = 仅金额
  source: 'rule_engine' | 'glm';
}

## 5. 离线优先架构

### 写入策略

```
所有写操作 → 先写本地 SQLite → 标记 sync_status = pending
                                     │
                              检测到网络可用时
                                     │
                              后台静默批量同步到 CloudBase
                                     │
                              成功 → sync_status = synced
                              失败 → 进入重试队列（指数退避）
```

### 离线/在线功能矩阵

| 功能 | 离线 | 在线 |
|------|------|------|
| 手动录入 | 完全可用 | 完全可用 |
| 文字记账（规则引擎） | 完全可用 | 完全可用 |
| 查看历史/统计 | 完全可用 | 完全可用 |
| 文字记账（GLM 兜底） | 不可用，提示手动选分类 | 可用 |
| 语音记账 | 不可用，提示无网络 | 可用 |
| 拍照识票 | 不可用，提示无网络 | 可用 |
| AI 自然语言查询 | 不可用，提示无网络 | 可用 |
| 云同步 | 队列等待 | 自动同步 |

## 6. 文字解析流程

```
用户输入: "午饭吃了25元"
     │
     ▼
┌──────────────────┐
│  本地规则引擎      │  ← 第一优先级，离线可用
│  正则 + 关键词匹配  │
└────────┬─────────┘
         │
    匹配成功？
    ├── 是 → 生成结构化数据 { type:expense, amount:25, category:餐饮 }
    │        → 展示确认卡片（金额/分类/账户可修改）
    │        → 用户确认 → 入库
    │
    └── 否 → 检查网络
             ├── 有网络 → 发给 GLM 解析 → 展示确认卡片 → 入库
             └── 无网络 → 提取出金额（如果有）
                          → 提示用户手动选择分类 → 入库
```

### 规则引擎匹配策略

1. **金额提取**：正则匹配数字 + 元/块/刀（USD）
2. **分类推断**：关键词映射表（见附录 A）
3. **收支判断**：默认支出；关键词命中"工资/收入/转入"时为收入
4. **置信度**：金额 + 分类都匹配到 → 高置信，直接展示确认卡片；只匹配到金额 → 低置信，需用户补充分类

### 确认卡片交互规则

规则引擎或 AI 解析后，在聊天中展示确认卡片（content_type = card）：

- **卡片内容**：金额、分类、账户、备注、日期，均可点击修改
- **确认**：用户点击"确认" → 创建 Transaction 入库，卡片变为"已记录"状态
- **取消**：用户点击"取消" → 卡片标记为 parse_status = 'error'，不创建 Transaction，显示"已取消"
- **超时**：卡片不过期，用户可随时确认或取消
- **已确认的账单**：可通过 transaction/[id] 页面编辑或删除（软删除）

## 8. 子项目开发路线图

9. **云同步 + 认证** — CloudBase 集成、冲突解决、用户登录

10. **数据导出** — CSV 生成、分享/保存

## 附录 B：设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 币种约束 | Transaction.currency 必须与 Account.currency 一致 | 避免汇率计算复杂度，P0 不做跨币种 |
| 软删除 | 所有表使用 deleted_at 而非硬删除 | 离线多设备同步需要传播删除操作 |
| 余额计算 | 查询时计算，不存储 balance 字段 | 避免余额与交易记录不一致 |
| 自定义分类 | 预留 is_system 字段，当前不开放 | 降低 P0 复杂度，避免未来 schema 迁移 |
| 子项目3拆分 | 3a 聊天框架 + 3b 确认卡片/手动录入 | 单个子项目范围过大，拆分后更易交付 |
